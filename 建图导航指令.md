# 建图与导航快速指令

> 地图目录：`/data/unitree/botbrain_ws/botbrain_ws/src/g1_pkg/maps/`，以下以 `floor1` 为例。
>
> 可视化统一为工作站本地运行的 **RViz2 + Zenoh**。建图与导航均不使用 Foxglove；机器人和工作站必须能访问 Zenoh TCP 端口 `7448`。完整界面操作见同目录 [rviz-guide.md](rviz-guide.md)。
>
> 必须从项目内的两份预设启动 RViz2：建图使用 `configs/g1_mapping_rviz2.rviz`，定位/导航使用 `configs/g1_nav_loc_rviz2.rviz`；预设已经配置正确的 Fixed Frame、话题和交互工具。
>
> 这是可视化迁移，不是修复 Foxglove 的 ROS 数据问题：FAST-LIO、定位、导航、话题和 TF 链路保持原样，RViz2 直接订阅同一套 ROS 2 数据。

> 机器人 Wi‑Fi IP 变化时，只替换 RViz 启动脚本最后的 IP 参数，例如 `bash tools/host_side/mapping_rviz2.sh 192.168.100.3`；不要修改 Unitree 内部 `192.168.123.x` 控制网。

## 现场常用

下面是已经完成本页“配置 PCD 输出/编译”后的日常入口。正常现场操作只需要这些服务命令和一条工作站 RViz2 命令；保存、修图、首建和故障排查继续看后文。

### 建图

机器人端：

```bash
cd /data/unitree/botbrain_ws
docker compose stop navigation localization
docker compose up -d zenoh bringup state_machine
FAST_LIO_MAPPING_PROFILE=default docker compose up -d --force-recreate fast_lio
```

长走廊把最后一行的 `default` 改为 `corridor`。保持机器人静止，看到 `IMU Initial Done` 后再移动。

工作站：

```bash
cd ~/Workspace/g1_botbrain_greeter
bash tools/host_side/mapping_rviz2.sh 192.168.100.3  # 改为当前 G1 IP
```

### 定位与导航

机器人端：

```bash
cd /data/unitree/botbrain_ws
scene=floor1
docker compose up -d zenoh bringup state_machine
# 同楼层且 FAST-LIO 已健康时使用：
bash tools/nav/select_map_scene.sh "$scene" --wait-ready --ready-timeout 300
# 冷启动、FAST-LIO 未运行/不健康或跨楼层时，将上一行替换为：
# bash tools/nav/select_map_scene.sh "$scene" --restart-fast-lio --wait-ready --ready-timeout 300
# 只有场景选择脚本返回 0 后才执行：

```

工作站：

```bash
cd ~/Workspace/g1_botbrain_greeter

```

`select_map_scene.sh` 会在导航前强制使用 FAST-LIO `default` 配置，避免长走廊建图模式带入导航。

## 一、建图

### 步骤 0：停止旧服务

```bash
cd /data/unitree/botbrain_ws
docker compose stop navigation localization fast_lio
```

### 步骤 1：配置 PCD 输出

修改 `/data/unitree/botbrain_ws/botbrain_ws/src/fast_lio/config/mid360.yaml`：

```yaml
map_file_path: "/botbrain_ws/src/g1_pkg/maps/floor1_scans.pcd"
pcd_save:
  pcd_save_en: true
```

如果有修改代码需要进行重新编译：

```bash
cd /data/unitree/botbrain_ws
docker compose up builder_base
```

### 步骤 2：启动传感器

```bash
cd /data/unitree/botbrain_ws
docker compose stop localization navigation
docker compose up -d zenoh bringup state_machine
# 普通场景使用 default；长走廊把 default 改成 corridor。
FAST_LIO_MAPPING_PROFILE=default docker compose up -d --force-recreate fast_lio

# 等待 IMU 初始化，机器人保持静止；满足后按 Ctrl+C 退出日志跟随
docker compose logs -f fast_lio | grep -E "IMU Initial|FAST_LIO_TIMING|FAST_LIO_GUARD|FAST_LIO_RANGE"
```

正常建图只需等待 `IMU Initial Done` 后移动。以下话题检查仅在 RViz 没有实时数据时执行：

```bash
docker exec -it g1_robot_fast_lio bash
source /opt/ros/humble/setup.bash && source /botbrain_ws/install/setup.bash
timeout 10 ros2 topic hz /livox/imu || true
timeout 10 ros2 topic hz /cloud_registered_1 || true
timeout 10 ros2 topic hz /Odometry_loc || true
```

### 步骤 2.1：工作站打开 RViz2 观察建图

> 在 **workstation** 执行。工作站只需安装一次 `ros-humble-rmw-zenoh-cpp` 和 `ros-humble-rviz2`；脚本会自动连接 Zenoh、刷新 ROS 2 图并打开建图预设。

```bash
cd ~/Workspace/g1_botbrain_greeter
bash tools/host_side/mapping_rviz2.sh 192.168.100.3  # 改为当前 G1 IP
```

RViz2 建图预设的 `Fixed Frame` 已是 `camera_init`，直接查看 `/cloud_registered_1` 和 `/accumulated_grid`。不要用 `/cloud_registered_body_1` 判断世界地图漂移。

长走廊只需把步骤 2 中的 `FAST_LIO_MAPPING_PROFILE=default` 改为 `FAST_LIO_MAPPING_PROFILE=corridor`，并确认启动日志显示对应 profile。该模式会保留更多门框/房间边缘点并实际过滤 20 米外输入，减少远处平行墙面的影响；当前工程没有位姿图回环，不能自动拉回已经写入的整张旧地图。经过开门房间时应减速或短暂停留，并左右转动观察房间角点；走廊尽头不要原地一次性转过 180 度，分段转向并确认 RViz 中门框、房间轮廓没有出现双层后再返回。

### 步骤 3：保存地图

保存 2D 栅格：

```bash
docker exec -it g1_robot_fast_lio bash
source /opt/ros/humble/setup.bash && source /botbrain_ws/install/setup.bash
ros2 run nav2_map_server map_saver_cli \
  -t /accumulated_grid --free 0.196 --occ 0.65 \
  -f /botbrain_ws/src/g1_pkg/maps/floor1
```

主动保存 PCD：

```bash
ros2 service call /map_save std_srvs/srv/Trigger '{}'
exit
```

必须看到 `success=true` 且消息包含 `saved <点数> points to <目标文件>` 后才继续；否则不要停止 FAST-LIO，先检查 `pcd_save_en` 和 `map_file_path`。

停止 FAST-LIO：

```bash
cd /data/unitree/botbrain_ws
docker compose stop -t 180 fast_lio
```

将mid360.yaml调回非建图状态：

```yaml
map_file_path: "/botbrain_ws/src/g1_pkg/maps/floor1_scans.pcd"
pcd_save:
  pcd_save_en: false
```

验证文件：

```bash
maps=/data/unitree/botbrain_ws/botbrain_ws/src/g1_pkg/maps
ls -lh "$maps/floor1_scans.pcd" "$maps/floor1.pgm" "$maps/floor1.yaml"
head -n 11 "$maps/floor1_scans.pcd" | grep -E "^(FIELDS|POINTS|DATA)"
```

### 步骤 4：PCD 高度校正（只做一次）

```bash
cd /data/unitree/botbrain_ws
maps=botbrain_ws/src/g1_pkg/maps
test ! -e "$maps/floor1_scans_raw.pcd" || {
  echo "ERROR: floor1_scans_raw.pcd 已存在，禁止覆盖或重复高度校正"; exit 1;
}
cp -a "$maps/floor1_scans.pcd" "$maps/floor1_scans_raw.pcd"

docker compose run --rm builder_base bash -lc '
  source /opt/ros/humble/setup.bash && source /botbrain_ws/install/setup.bash
  python3 /botbrain_ws/install/g1_pkg/lib/g1_pkg/shift_pcd_z.py \
    /botbrain_ws/src/g1_pkg/maps/floor1_scans.pcd 1.247
'
```

校正后将 `mid360.yaml` 的 `pcd_save_en` 改为 `false` 并重新编译。

### 步骤 5：确认成品地图场景

```bash
scene=floor1
maps=/data/unitree/botbrain_ws/botbrain_ws/src/g1_pkg/maps
test -s "$maps/${scene}_scans.pcd" \
  && test -s "$maps/${scene}.yaml" \
  && test -s "$maps/${scene}.pgm" || {
    echo "ERROR: $scene 的地图三文件缺失或为空"; exit 1;
  }
sed -n '1,10p' "$maps/${scene}.yaml"
```

标准命名固定为 `<scene>_scans.pcd + <scene>.yaml + <scene>.pgm`，YAML 的 `image` 必须指向同场景 PGM。日常通过 `select_map_scene.sh <scene>` 选择整套地图，不再创建 `scans.pcd`、`accumulated.yaml` 软链接。`MAP_SCENE` 是脚本创建 localization 容器时使用的内部参数，单独 `export MAP_SCENE=...` 不会切换已运行容器。完整三文件检查见 `机器人项目run.md` 的 2.3。

### 步骤 6：建图完成后查看效果

启动 `bringup + fast_lio + localization`，但不启动 `navigation`。场景选择器会同时检查 FAST-LIO、定位置信度、`/scan`、Unitree 速度里程计、TF、连续 ICP 质量和关键话题 publisher 数量。

```bash
cd /data/unitree/botbrain_ws
scene=floor1

docker compose up -d zenoh bringup state_machine
bash tools/nav/select_map_scene.sh "$scene" --restart-fast-lio --wait-ready --ready-timeout 300

# 返回前会直接打印 3 行 ICP accepted: fitness=... rmse=...
# 且 preflight 必须显示 twist_source=unitree；返回 0 才表示验收通过
# 以下只用于继续观察后续定位变化，按 Ctrl+C 不会停止后台服务
docker compose logs -f localization | \
  grep --line-buffered -E "Map selection:|Localization ready|ICP: accepted|ERROR|FATAL"
```

在 workstation 执行下面一条命令打开导航预设：

```bash
cd ~/Workspace/g1_botbrain_greeter
bash tools/host_side/g1_nav_loc_rviz2.sh 192.168.100.3
```

- `/localization_ready=false` 时，继续使用建图预设（Fixed Frame=`camera_init`）确认 `/cloud_registered_1`、`/path_1` 有实时数据。
- `/map` 出现不等于定位完成；等日志出现 `Localization ready` 或 `/localization_ready=true` 后，再打开导航预设。
- 导航预设的 Fixed Frame 已设为 `map`，包含 `/map`、`/pcd_map`、`/scan`、`/g1_robot/plan` 与 `/g1_robot/goal_pose` 工具。
- 俯视检查 2D 地图与 PCD 墙体轮廓，侧视检查 PCD 高度、倾斜和实时点云是否重合。
- ready 前以 `map` 为 Fixed Frame 时，实时点云、Path 和 Scan 可能因缺少 `map -> odom` TF 而消失；这不是话题丢失。

机器人静止、直行和左右转弯后，`/cloud_registered_1` 都应与 `/pcd_map` 和 `/map` 保持重合。

2D 孤立黑点可进入下一节修图；如果 PCD 已经出现双墙、倾斜或明显漂移，不能只修 PGM，应重新检查或重新建图。

## 三、定位与导航启动

### 启动前编译

```bash
cd /data/unitree/botbrain_ws
docker compose up builder_base
```

### 启动定位

```bash
cd /data/unitree/botbrain_ws
scene=floor1  # 改为机器人当前所在场景，例如 ug、floor4
# 基础服务已运行时可重复执行，无需 stop/rm。
docker compose up -d zenoh bringup state_machine
```

下面两条定位命令二选一。同楼层且 FAST-LIO 已经健康时，使用较快的方式：

```bash
bash tools/nav/select_map_scene.sh "$scene" \
  --wait-ready --ready-timeout 300
```

机器人冷启动、FAST-LIO 未运行/不健康或跨楼层时，使用完整方式：

```bash
bash tools/nav/select_map_scene.sh "$scene" \
  --restart-fast-lio --wait-ready --ready-timeout 300
```

需要观察定位日志时执行：

```bash
docker compose logs -f localization | \
  grep --line-buffered -E "Map selection:|Localization ready|ICP: accepted|ERROR|FATAL"
```

`--ready-timeout 300` 是导航验收的最长等待上限，不是固定暂停 300 秒。直接执行 `docker compose up -d` 看起来更快，是因为命令在容器创建后立即返回，IMU 初始化、FPFH/RANSAC 和 ICP 仍在后台进行，并不代表已经可以导航。

- 机器人冷启动、FAST-LIO 未运行/不健康或真正跨楼层：使用上面带 `--restart-fast-lio` 的完整命令。
- 同一楼层重载地图，且当前 FAST-LIO 已经健康：改用 `bash tools/nav/select_map_scene.sh "$scene" --wait-ready --ready-timeout 300`，跳过 FAST-LIO 重建和 IMU 重新初始化。
- 为了导航安全，不要通过删除 `--wait-ready` 后立即启动 Navigation 来“提速”；只有脚本返回 `0` 才启动 Navigation。

场景选择脚本成功返回后，在工作站另开终端启动定位/导航 RViz2：

```bash
cd ~/Workspace/g1_botbrain_greeter
bash tools/host_side/g1_nav_loc_rviz2.sh 192.168.37.204
```

- RViz2 预设的 Fixed Frame 必须为 `map`
- 先等待 FPFH/RANSAC 自动初始化；只有持续失败时才在 RViz2 工具栏选择 `2D Pose Estimate` 发送 `/initialpose`，并确认 `Manual relocalization applied`
- 场景选择器返回前必须直接打印连续 3 行 `ICP accepted:`（fitness>0.50, rmse<=0.30），且 preflight 显示 `twist_source=unitree`；否则不能启动导航
- 若提示 `navigation install space is missing or stale`，先执行本节“启动前编译”，不能继续使用旧 install 产物

### 启动导航

只有上一步场景选择器返回 `0` 时才执行；返回非 `0` 时 Navigation 必须保持停止。

```bash
cd /data/unitree/botbrain_ws
docker compose --profile navigation up -d --force-recreate navigation
docker compose logs -f navigation
```

看到 `Managed nodes are active` 且没有 `ERROR` 后按 `Ctrl+C` 退出日志跟随；后台 Navigation 不会停止。详细 lifecycle 检查仅在启动失败时参考 `机器人项目run.md`。

如果定位阶段还没有打开 RViz2，在工作站执行；如果已打开，直接继续使用同一个窗口：

```bash
cd ~/Workspace/g1_botbrain_greeter
bash tools/host_side/g1_nav_loc_rviz2.sh 192.168.100.3
```

确认 Fixed Frame=`map`、`Map`=`/map`、静态点云=`/pcd_map`、实时激光=`/scan`。仅当 `Localization ready`、Nav2 生命周期检查均通过后，才在工具栏选择 `2D Goal Pose` 并在地图上点击拖拽目标朝向；预设会向 `/g1_robot/goal_pose` 发布目标。需要人工重定位时选择 `2D Pose Estimate`，同样在 Fixed Frame=`map` 下点击拖拽；不要在 `camera_init` 下发送 `/initialpose`。RViz 图层和视角调整见 `rviz-guide.md`，现场不需要手动执行 `source`、`ros2 daemon` 或 `rviz2 -d` 命令。

### 点位记录与导航

```bash
docker exec -it g1_robot_navigation bash
source /opt/ros/humble/setup.bash && source /botbrain_ws/install/setup.bash

ros2 run bot_navigation waypoint_recorder.py record floor1_office #记录点位
ros2 run bot_navigation waypoint_recorder.py list            # 查看点位
ros2 run bot_navigation waypoint_recorder.py delete floor1_old # 删除指定点位
ros2 run bot_navigation waypoint_recorder.py list            # 删除后确认
ros2 run bot_navigation waypoint_navigator.py floor1_office #单点导航
ros2 run bot_navigation waypoint_navigator.py floor1_kitchen floor1_office floor1_home #多点导航,--loop为循环
```

> `delete` 只删除名称完全一致的一个点位；名称不存在时会报错，不会修改文件。删除后无需重新编译或重启 Navigation，下一次执行 `waypoint_navigator.py` 时会重新读取文件。
>
> 点位文件：`/data/unitree/botbrain_ws/botbrain_ws/src/bot_navigation/nav_waypoints.yaml`。文件内按场景分区；`select_map_scene.sh` 成功切图后会同步当前场景，上述记录、查看、删除和导航命令都只操作当前场景的点位。导航前还会核对 map_server 实际场景，并拒绝发送明显落在占用栅格上的目标。
>
> 首次部署先备份该文件，不要用仓库默认文件覆盖现场点位。旧顶层 `waypoints:` 格式会在选中正确场景后，于首次导航或 `record`/`delete` 时自动迁移到该场景。

## 四、多楼层地图切换

以下把目标场景写成 `floor4`。机器人必须先到达目标楼层、取消活动导航目标并停稳；禁止运动中热切图。

```bash
cd /data/unitree/botbrain_ws
scene=floor4

# 先按步骤 5 检查 floor4_scans.pcd、floor4.yaml、floor4.pgm
# 跨楼层使用 --restart-fast-lio，在目标层重建 camera_init/odom 基准。
# 脚本会校验三文件、等待 IMU Initial Done、重建 localization，
# 并等待 scan、Unitree twist、ready、confidence、TF、连续 ICP 和 publisher 唯一性全部通过。
bash tools/nav/select_map_scene.sh "$scene" \
  --restart-fast-lio --wait-ready --ready-timeout 300
docker compose logs -f localization | \
  grep --line-buffered -E "Map selection:|Global registration|Global candidate|Localization ready|ICP: accepted|ERROR|FATAL"

# 场景选择器返回 0 后，最后重建导航
docker compose --profile navigation up -d --force-recreate navigation
```

`select_map_scene.sh` 可在容器正在运行时直接调用。同一楼层换图用 `bash tools/nav/select_map_scene.sh aitech --wait-ready --ready-timeout 300`；机器人实际跨楼层用 `bash tools/nav/select_map_scene.sh floor4 --restart-fast-lio --wait-ready --ready-timeout 300`。脚本会校验三文件、删除旧 localization/navigation 容器、显式传入 `MAP_SCENE`，并硬性确认容器环境与 `Map selection:` 日志一致。不要只调用 `map_server/load_map`，也不要用 `docker compose restart localization` 切层。

该流程支持每层独立导航和停车切层，不支持 Nav2 直接规划跨楼层连续路径，也不会自动识别楼层。跨层任务需由上层状态机拆成“到电梯 -> 乘梯 -> 目标层重建 FAST-LIO/定位 -> 使用目标层点位”。

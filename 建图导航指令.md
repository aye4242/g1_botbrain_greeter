# 建图与导航快速指令

> 地图目录：`/data/unitree/botbrain_ws/botbrain_ws/src/g1_pkg/maps/`，以下以 `floor1` 为例。
>
> 可视化统一为工作站本地运行的 **RViz2 + Zenoh**。建图与导航均不使用 Foxglove；机器人和工作站必须能访问 Zenoh TCP 端口 `7448`。完整界面操作见同目录 [rviz-guide.md](rviz-guide.md)。
>
> 必须从项目内的两份预设启动 RViz2：建图使用 `configs/g1_mapping_rviz2.rviz`，定位/导航使用 `configs/g1_nav_loc_rviz2.rviz`；预设已经配置正确的 Fixed Frame、话题和交互工具。
>
> 这是可视化迁移，不是修复 Foxglove 的 ROS 数据问题：FAST-LIO、定位、导航、话题和 TF 链路保持原样，RViz2 直接订阅同一套 ROS 2 数据。

> 机器人 Wi‑Fi IP 变化时，只替换 RViz 启动脚本最后的 IP 参数，例如建图使用 `bash tools/host_side/mapping_rviz2.sh 192.168.100.3`，导航使用 `bash tools/host_side/g1_nav_loc_rviz2.sh 192.168.100.3`；不要修改 Unitree 内部 `192.168.123.x` 控制网。

## 现场常用

下面是部署完成后的日常入口。建图默认直接启动 Compose 服务，命令立即返回；随后通过 `docker compose ps` 和 `docker compose logs` 观察状态。保存、修图和故障排查继续看后文。

### 建图

机器人端：

```bash
cd /data/unitree/botbrain_ws
scene=floor1
profile=default  # 长走廊改为 corridor

docker compose stop localization navigation
docker compose up -d zenoh bringup state_machine
touch "botbrain_ws/src/g1_pkg/maps/.${scene}_mapping_started"

FAST_LIO_START_DELAY_SEC=0 \
FAST_LIO_MAPPING_MODE=true \
FAST_LIO_MAPPING_SAVE=true \
FAST_LIO_MAP_FILE="/botbrain_ws/src/g1_pkg/maps/${scene}_scans.pcd" \
FAST_LIO_MAPPING_PROFILE="$profile" \
docker compose up -d --force-recreate fast_lio

docker compose ps zenoh bringup state_machine fast_lio
docker compose logs -f fast_lio
```

`docker compose up -d` 会立即返回，不需要等待检查脚本。`docker compose logs -f fast_lio` 中看到 `IMU Initial Done`、连续 `[FAST_LIO_TIMING] ok=true`，并在 RViz2 看到两类点云和 `/accumulated_grid` 后再移动；按 `Ctrl+C` 只退出日志跟随，不会停止服务。服务方式不会自动备份同名地图：优先使用新场景名；需要重建旧场景时，先备份 `<scene>_scans.pcd/.pgm/.yaml`。

**长走廊 + 移动 Wi-Fi 现场：** 必须使用 `profile=corridor`。实时栅格已限制为 0.5 Hz，且每 3 帧云处理一次，以避免大地图的 Python 序列化占满 Jetson/Zenoh；这不降低 FAST-LIO 本身的 10 Hz 建图频率。若出现 `[FAST_LIO_GUARD] output latched unhealthy`，立即停止移动并重建 `fast_lio`，不要继续等待。

工作站：

```bash
cd ~/Workspace/g1_botbrain_greeter
bash tools/host_side/mapping_rviz2.sh 192.168.37.204  # 改为当前 G1 IP
```

该脚本不是直接打开一个空窗口：它会先从工作站侧实际收到一次
`/cloud_registered_1`，并使用 FAST-LIO 的 BEST_EFFORT QoS 探测
`/cloud_registered_body_1`。世界点云是硬性门槛；Body 点云是诊断图层，若
Zenoh 刚启动时发现延迟，脚本会警告但仍打开 RViz2，预置的 Body 显示会继续
自动发现，避免把正常建图误判为失败。若 Body 图层在 30 秒后仍为 Warm，才检查
`publish.scan_bodyframe_pub_en` 和 FAST-LIO 日志。不能靠手工 Add 绕过世界点云缺失。
建图预设同时提供明亮的当前世界扫描和最近 5 分钟世界点云历史，效果与旧
FAST-LIO/Foxglove 建图视图一致，并保留橙色 Body 点云。

### 定位与导航（必须按顺序）

机器人端先停止 Navigation，只启动基础服务和定位选择器。`localization` 没有漏掉：它是 Compose 服务，由 `select_map_scene.sh` 在内部使用当前 `$scene` 自动创建并启动；不要在脚本外再启动第二份 localization：

```bash
cd /data/unitree/botbrain_ws
scene=floor1
docker compose stop navigation
docker compose up -d zenoh bringup state_machine
# 同楼层且 FAST-LIO 已健康时使用：
bash tools/nav/select_map_scene.sh "$scene" --wait-ready --ready-timeout 300
# 刚结束建图、冷启动、FAST-LIO 未运行/不健康或跨楼层时，将上一行替换为：
# bash tools/nav/select_map_scene.sh "$scene" --restart-fast-lio --wait-ready --ready-timeout 300
```

脚本内部实际会执行等价于下面的服务命令，因此运行过程中 `docker compose ps localization` 应显示 `Up`：

```bash
MAP_SCENE="$scene" LOCALIZATION_START_DELAY_SEC=0 \
docker compose --profile navigation up -d \
  --force-recreate --no-deps localization
```

最终运行关系是 `fast_lio + localization + navigation` 三个服务同时存在，但启动顺序必须是：先 FAST-LIO，再 localization 匹配成功，最后 Navigation。不能把三者无条件同时拉起。

场景选择器会加载 PCD/PGM 并验证地图发布者；只有使用 `--restart-fast-lio` 时才会重建 FAST-LIO。看到这一行后，不需要等脚本最终返回，就在工作站打开导航 RViz2：

```text
RVIZ POINT CLOUD READY: /pcd_map is publishing for scene '<scene>'.
```

```bash
cd ~/Workspace/g1_botbrain_greeter
bash tools/host_side/g1_nav_loc_rviz2.sh 192.168.37.204
```

导航脚本同样会先收到 `/pcd_map` 才打开窗口；如果 localization 尚未启动，
它会明确退出并提示改用建图脚本，不再打开一个全部 Warm 的导航窗口。

日志中的节点名 `navigation_preflight` 只是“导航前检查器”，不是让你启动 Navigation。看到下面这种输出时，Navigation 必须仍保持停止：

```text
Waiting for navigation inputs: ... ready=false confidence=0.000 ... tf=false
```

RViz2 在定位完成前必须同时看到高亮蓝色成品点云 `/pcd_map` 和紫色实时扫描预览 `/scan2map`。`/scan2map` 在尚无候选时先按当前未验证的 `map <- odom` 猜测放置，让机器人点云始终可见；算法得到全局/局部候选后会更新到候选位置。它只是可视化，不代表定位通过，也不会放行 Navigation。绿色 `/cloud_registered_1` 和橙色 `/cloud_registered_body_1` 此时可能仍显示 Warm，因为尚无经验证的 `map -> camera_init/body` TF；定位成功后它们会自动出现，无需 Add。

如果连蓝色 `/pcd_map` 都没有，禁止盲发 `2D Pose Estimate`，也不要启动 Navigation；先确认场景选择器已经打印 `RVIZ POINT CLOUD READY` 且 `docker compose ps localization` 为 `Up`。它表示 `/scan` 和里程计可能已正常，但机器人尚未在地图中定位。先等待自动 FPFH/RANSAC；如果日志出现 `Global initialization completed 6 multi-scale attempts without a verified pose`，或者约 60～90 秒仍无法从 `1/3` 到 `3/3`，就在蓝色 PCD/二维地图上按机器人真实位置，使用 `2D Pose Estimate` 点击并拖出实际朝向。确认日志依次出现：

```text
Manual relocalization applied
Local localization initialization succeeded
Localization ready: verified map->odom is now available
Navigation preflight passed
Scene '<scene>' localization is ready for navigation.
```

只有场景选择脚本返回 0 后才执行 Navigation；也就是最后一行出现、`select_map_scene.sh` 返回到 shell 且退出码为 `0` 后，才执行：

```bash
docker compose --profile navigation up -d --force-recreate navigation
docker compose logs -f navigation
docker compose ps fast_lio localization navigation
```

也可以直接把“匹配成功后启动 Navigation”连成一条命令；前半段任何检查失败时，`&&` 后面的 Navigation 不会启动：

```bash
bash tools/nav/select_map_scene.sh "$scene" \
  --restart-fast-lio --wait-ready --ready-timeout 300 && \
docker compose --profile navigation up -d --force-recreate navigation
```

如果 Navigation 已经提前启动并持续打印 `ready=false`，先执行 `docker compose stop navigation`，再重新按本节顺序操作。启动 Navigation 不会代替定位，也不会生成缺失的 `map -> odom` TF。`select_map_scene.sh` 会在导航前强制使用 FAST-LIO `default` 配置，避免长走廊建图模式带入导航。

## 一、建图

### 步骤 0：停止定位与导航

```bash
cd /data/unitree/botbrain_ws
docker compose stop localization navigation
```

建图期间保持这两个服务停止，避免旧地图和定位 TF 与新地图叠加。无需手动 `rm` 容器。

### 步骤 1：首次部署代码后编译

只有代码或 launch 有更新时才编译；日常建图不再修改 `mid360.yaml`：

```bash

```

### 步骤 2：直接启动建图服务


看到 `IMU Initial Done`、连续 `[FAST_LIO_TIMING] ok=true`，并在 RViz2 看到世界点云、Body 点云和累计栅格后才能移动。`Ctrl+C` 只退出 `logs -f`，后台服务继续运行。

### 步骤 2.1：工作站打开 RViz2 观察建图

> 在 **workstation** 执行。工作站只需安装一次 `ros-humble-rmw-zenoh-cpp` 和 `ros-humble-rviz2`；脚本会自动连接 Zenoh、刷新 ROS 2 图并打开建图预设。


RViz2 建图预设的 `Fixed Frame` 已是 `camera_init`。`/cloud_registered_1`

可见，`world history (5 min)` 保留最近 300 秒用于观察建图轮廓；橙色
`/cloud_registered_body_1`、累计栅格、里程计、轨迹和 TF 也均已预置。
不要用橙色 Body 点云判断世界地图漂移。

如果工作站为 Intel/Mesa，并出现

### 步骤 5：确认成品地图场景

```bash
scene=floor1
maps=/data/unitree/botbrain_ws/botbrain_ws/src/g1_pkg/maps
test -s "$maps/${scene}_scans.pcd" \
  && test -s "$maps/${scene}.yaml" \
  && test -s "$maps/${scene}.pgm" || {
    echo "ERROR: $scene 的地图三文件缺失或为空"; exit 1;
  }
sed -n '1,10p' "$maps/${scene}.yaml"
```

标准命名固定为 `<scene>_scans.pcd + <scene>.yaml + <scene>.pgm`，YAML 的 `image` 必须指向同场景 PGM。日常通过 `select_map_scene.sh <scene>` 选择整套地图，不再创建 `scans.pcd`、`accumulated.yaml` 软链接。`MAP_SCENE` 是脚本创建 localization 容器时使用的内部参数，单独 `export MAP_SCENE=...` 不会切换已运行容器。完整三文件检查见 `机器人项目run.md` 的 2.3。

### 步骤 6：建图完成后查看效果

启动 `bringup + fast_lio + localization`，但不启动 `navigation`。场景选择器会同时检查 FAST-LIO、定位置信度、`/scan`、Unitree 速度里程计、TF、连续 ICP 质量和关键话题 publisher 数量。

```bash
cd /data/unitree/botbrain_ws
scene=floor1

docker compose up -d zenoh bringup state_machine
bash tools/nav/select_map_scene.sh "$scene" --restart-fast-lio --wait-ready --ready-timeout 300

# 返回前会直接打印 3 行 ICP accepted: fitness=... rmse=...
# 且 preflight 必须显示 twist_source=unitree；返回 0 才表示验收通过
# 以下只用于继续观察后续定位变化，按 Ctrl+C 不会停止后台服务
docker compose logs -f localization | \
  grep --line-buffered -E "Map selection:|Localization ready|ICP: accepted|ERROR|FATAL"
```

在 workstation 执行下面一条命令打开导航预设：

```bash
cd ~/Workspace/g1_botbrain_greeter
bash tools/host_side/g1_nav_loc_rviz2.sh 192.168.37.204
```

- `/localization_ready=false` 时，继续使用建图预设（Fixed Frame=`camera_init`）确认两类点云和 `/path_1` 有实时数据；不要用导航预设是否显示点云判断建图是否成功。
- `/map` 出现不等于定位完成；等日志出现 `Localization ready` 或 `/localization_ready=true` 后，再打开导航预设。
- 导航预设的 Fixed Frame 已设为 `map`，自动包含 `/map`、`/pcd_map`、`/cloud_registered_1`、`/cloud_registered_body_1`、`/scan`、`/g1_robot/plan`、TF、可选 global/local costmap 与 `/g1_robot/goal_pose` 工具，无需手动 Add。
- 修改过预设后先关闭旧 RViz2 再重开；若左侧仍出现 `RobotModel (G1)`，说明当前是旧窗口，当前预设没有 RobotModel。
- 俯视检查 2D 地图与 PCD 墙体轮廓，侧视检查 PCD 高度、倾斜和实时点云是否重合。
- ready 前以 `map` 为 Fixed Frame 时，实时点云、Path 和 Scan 可能因缺少 `map -> odom` TF 而消失；这不是话题丢失。

机器人静止、直行和左右转弯后，`/cloud_registered_1` 都应与 `/pcd_map` 和 `/map` 保持重合。

2D 孤立黑点可进入下一节修图；如果 PCD 已经出现双墙、倾斜或明显漂移，不能只修 PGM，应重新检查或重新建图。

### 启动定位

```bash
cd /data/unitree/botbrain_ws

# 基础服务已运行时可重复执行，无需 stop/rm。

```

此阶段只允许 `fast_lio + localization` 工作，`navigation` 必须停止。`localization` 本身就是 Compose 服务，接下来的 `select_map_scene.sh` 会按 `$scene` 在内部执行 `docker compose ... up -d localization`；不需要也不能额外启动第二份。下面命令输出中的 `navigation_preflight` 是定位/导航输入检查器，不是 Navigation 服务，也不是要求再执行一次 `docker compose up navigation`。

下面两条定位命令二选一。同楼层且 FAST-LIO 已经健康时，使用较快的方式：

```bash
bash tools/nav/select_map_scene.sh "$scene" \
  --wait-ready --ready-timeout 300
```

`--ready-timeout 300` 是导航验收的最长等待上限，不是固定暂停 300 秒。
Compose 中 FAST-LIO、localization 和 Navigation 的固定 25/30 秒 sleep 已取消；
节点会立即启动并依靠真实数据门禁。`docker compose up -d` 只是容器创建后立即
返回，IMU 初始化、FPFH/RANSAC 和 ICP 仍在后台进行，并不代表已经可以导航。
定位节点会在准备全局 FPFH 特征前先发布 `/pcd_map`，所以成品点云不应再被
定位计算挡住。FAST-LIO 重启检查读取最近 5 次 timing：允许其中 1 个被保护逻辑
安全丢弃的坏帧，但要求至少 `4/5` 为 `ok=true`、最新一次为 `ok=true`，并实际
读取里程计和世界点云。

- 刚结束建图、机器人冷启动、FAST-LIO 未运行/不健康或真正跨楼层：使用上面带 `--restart-fast-lio` 的完整命令。脚本会拒绝把仍启用 PCD 保存或 corridor profile 的建图容器直接带入导航。
- 同一楼层重载地图，且当前 FAST-LIO 已经健康：改用 `bash tools/nav/select_map_scene.sh "$scene" --wait-ready --ready-timeout 300`，跳过 FAST-LIO 重建和 IMU 重新初始化。
- 为了导航安全，不要通过删除 `--wait-ready` 后立即启动 Navigation 来“提速”；只有脚本返回 `0` 才启动 Navigation。
- `ready=false confidence=0.000 tf=false` 表示定位未完成。若出现 `Global initialization completed 6 multi-scale attempts without a verified pose`，或 60～90 秒仍只有 `1/3`，在 RViz2 使用 `2D Pose Estimate`；不要继续等 Navigation 容器解决定位。
- 定位前必须先看到高亮蓝色 `/pcd_map` 和持续刷新的紫色 `/scan2map`。紫色层首先是未验证的实时扫描预览，得到定位候选后更新到候选位置；只用于查看和辅助 `2D Pose Estimate`，不表示定位成功。绿色/橙色实时点云要等 `map` TF 验证成功才会脱离 Warm，这是防止未验证变换被导航误用。
- 如果蓝色 `/pcd_map` 也完全没有，检查 `docker compose ps localization`；没有 `g1_robot_localization` 就说明场景选择器尚未加载到地图阶段、已经失败或已被中断，不能靠 RViz 自己补出地图。

### 启动导航

只有上一步明确打印 `Scene '<scene>' localization is ready for navigation.`、场景选择器返回到 shell 且退出码为 `0` 时才执行；仍在打印 `Waiting for navigation inputs` 就表示不能启动。返回非 `0` 时 Navigation 必须保持停止。

```bash

docker compose --profile navigation up -d --force-recreate navigation
docker compose logs -f navigation
```

Navigation 不再先盲等 30 秒；启动后立即执行独立 preflight。场景选择器已经
成功时通常数秒内进入 Nav2；若绕过场景选择器直接启动，preflight 最多等待
60 秒后退出，绝不会用未验证的 `map` TF 开始运动。


`LocoState not available`。这是机器人本体 locomotion 服务没有响应，不是 RViz、
Zenoh 或 Nav2 可视化问题；即使 `192.168.123.161` 能 ping 通，也不能在该状态下
验收导航运动。先恢复机器人本体运动服务/模式，再发送 `2D Goal Pose`。

看到 `Managed nodes are active` 且没有 `ERROR` 后按 `Ctrl+C` 退出日志跟随；后台 Navigation 不会停止。详细 lifecycle 检查仅在启动失败时参考 `机器人项目run.md`。

## 四、多楼层地图切换

以下把目标场景写成 `floor4`。机器人必须先到达目标楼层、取消活动导航目标并停稳；禁止运动中热切图。

```bash
cd /data/unitree/botbrain_ws
scene=floor4

# 先按步骤 5 检查 floor4_scans.pcd、floor4.yaml、floor4.pgm
# 跨楼层使用 --restart-fast-lio，在目标层重建 camera_init/odom 基准。
# 脚本会校验三文件、等待 IMU Initial Done、重建 localization，
# 并等待 scan、Unitree twist、ready、confidence、TF、连续 ICP 和 publisher 唯一性全部通过。
bash tools/nav/select_map_scene.sh "$scene" \
  --restart-fast-lio --wait-ready --ready-timeout 300
docker compose logs -f localization | \
  grep --line-buffered -E "Map selection:|Global registration|Global candidate|Localization ready|ICP: accepted|ERROR|FATAL"

# 场景选择器返回 0 后，最后重建导航
docker compose --profile navigation up -d --force-recreate navigation
```

`select_map_scene.sh` 可在容器正在运行时直接调用。同一楼层换图用 `bash tools/nav/select_map_scene.sh aitech --wait-ready --ready-timeout 300`；机器人实际跨楼层用 `bash tools/nav/select_map_scene.sh floor4 --restart-fast-lio --wait-ready --ready-timeout 300`。脚本会校验三文件、删除旧 localization/navigation 容器、显式传入 `MAP_SCENE`，并硬性确认容器环境与 `Map selection:` 日志一致。不要只调用 `map_server/load_map`，也不要用 `docker compose restart localization` 切层。

该流程支持每层独立导航和停车切层，不支持 Nav2 直接规划跨楼层连续路径，也不会自动识别楼层。跨层任务需由上层状态机拆成“到电梯 -> 乘梯 -> 目标层重建 FAST-LIO/定位 -> 使用目标层点位”。
